Framework Choice: Decoupled Discrete-Event SimulatorFor this scheduling engine, I have chosen to implement a Discrete-Event Simulator with a Pluggable Scoring System.Why it fits the problem:Exact solvers (like MILP) can be overly complex to implement from scratch in a few days and rely on heavy third-party libraries. Deep Learning is massive over-engineering.A timeline-based simulator evaluates the state of the network at discrete intervals (or events). When a decision needs to be made (e.g., multiple buses arrive at a station), the engine calls an isolated scoring function.This perfectly satisfies the requirement for scalability and extensibility. Adding a new rule simply means adding one new line to the scoring function, leaving the core simulation engine entirely untouched.2. Data Modeling & Foresight (The Data Store)The assignment emphasizes that the "scenario IS your data structure". To ensure the constraints are not hardcoded, the entire state of the world is defined within a JSON file for each scenario. The Streamlit app acts as a dumb terminal that simply reads this data store and executes it.  This design anticipates future changes without requiring any code rewrites:  Anticipated ChangeHow the Data Structure Handles It (Zero Code Changes)Adding a new stationSimply add a new node object to the route array in the JSON file.Doubling chargers at a stationChange the charger_count integer attribute on a specific station object.  Different bus modelsModify the max_range (currently 240) and charge_time (currently 25) attributes on individual bus objects.Priority busesIntroduce a priority_multiplier field to the bus object, which the scoring function can read.  Driver shift limitsAdd a max_drive_time attribute to the global configuration block of the JSON.  3. Handling Tunable WeightsThe weights for the three soft rules (individual wait times, operator fleet smoothness, overall network time) are stored at the top level of the JSON scenario file.Changing a weight is a trivial data edit in one obvious place. The scoring engine reads these weights dynamically at runtime:
final_score = (individual_score * weights.individual) + (operator_score * weights.operator) + (network_score * weights.overall)   Implementation Instructions (Your Build Plan)Follow these phases sequentially to build the project:Phase 1: Define the Schema: Create scenario_1.json. Define the global_config (weights, constants), route (segments and distances), stations (locations and charger counts), and fleet (buses, operators, directions, departure times).Phase 2: Build the Core Engine: Write a Python class that steps through time. Track the state of every bus (location, battery level) based on a consistent speed. Enforce the hard rule: a bus must charge if it cannot reach the next station.  Phase 3: Build the Scoring Heuristic: Create a separate function for the pluggable scoring system. When buses queue for a charger, evaluate them based on the individual, operator, and overall heuristics multiplied by the scenario's weights.Phase 4: Streamlit UI Integration: Keep the UI bare-bones as requested. Implement a dropdown to select the JSON file. Render three views: a display of the raw JSON input, a per-bus timetable, and a per-station charging queue order.
